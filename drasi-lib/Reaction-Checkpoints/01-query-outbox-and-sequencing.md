# Query Outbox and Sequencing

> **Parent**: [Reaction Recovery Overview](./00-reaction-recovery-overview.md)

---

## 1. Result Sequence Counter

A monotonic `u64` counter (`last_result_seq`) tracking emitted query results. Maintained in memory for all queries, and persisted in the `stream_state` CF for persistent queries.

- Increments only when diffs are non-empty.
- Applies equally to source-event diffs and time-triggered `process_due_futures` diffs.
- Uses `config_hash` verification on startup: matching hash resumes from persisted sequence, mismatch resets to `0`.

## 2. In-Memory `QueryOutputState`

All read operations (`fetch_snapshot`, `fetch_outbox`) are served from this unified struct behind an `RwLock`:

```rust
pub struct QueryOutputState {
    /// Live result set, keyed by row_signature for O(1) diff application.
    pub results: HashMap<u64, serde_json::Value>,
    /// The result sequence number the snapshot reflects.
    pub as_of_sequence: u64,
    /// Ring buffer of recent QueryResults (max configurable entries).
    pub outbox: VecDeque<QueryResult>,
}
```

**Write Path (Query Processor):**
After a successful DB commit, a write lock is acquired to apply the non-empty diffs (O(1) dictionary updates), increment `as_of_sequence`, and push/pop the `outbox` ring buffer.

**Startup (Crash Recovery):**
For persistent queries, `QueryOutputState` is hydrated by reading the `live_results` CF, `outbox` CF, and `stream_state` sequence upon initialization.

## 3. Persistent Storage Layout

Persistent queries write to three dedicated storage namespaces (Column Families in RocksDB, Hash Maps or Key Prefixes in Garnet) owned by `drasi-lib`. These are kept separate from core's graph storage (`elements`, `slots`, `metadata`, etc.) and are written exclusively within the outer transaction:

1. **`stream_state`**: Stores scalar orchestrator metadata.
   - `result_sequence` (u64): The highest emitted result sequence. (This also serves as the `as_of_sequence` for the live snapshot).
   - `source_sequence:{id}` (u64): The upstream checkpoint for each source.
   - `config_hash` (u64): Used to invalidate stale indexes on query updates.
2. **`outbox`**: The ring buffer.
   - **Key**: `result_sequence` (u64, big-endian).
   - **Value**: Serialized `QueryResult`. On append, if capacity > N, the query executes a `Delete` on the smallest keys.
3. **`live_results`**: The persistent analog to the `QueryOutputState.results` map.
   - **Key**: `row_signature` (u64, big-endian).
   - **Value**: Serialized row data. Updates are identical to the HashMap logic (`Add` = put, `Delete` = delete, `Update` = delete + put).

**Why distinct namespaces?** 
Isolation prevents the massive volume of delete tombstones generated by the FIFO `outbox` from degrading the read performance of `live_results`. It also cleanly separates lib's orchestration state from core's graph data, allowing targeted storage engine optimizations for each unique access pattern (e.g., scalar overwrites vs. sliding ring buffer vs. random-access KV).

Note: The old `snapshot_sequence` is perfectly in sync with `result_sequence` in this architecture because the live_results state is always atomically updated alongside the outbox append.

## 4. Extended Transaction Scope

Both `process_source_change` and `process_due_futures` execution paths are wrapped in the outer transaction:

```text
Lib:    session_control.begin()
Core:     ...index writes...
Lib:    checkpoint_writer.stage_checkpoint(src, seq) 
Lib:    if !diffs.is_empty():
          checkpoint_writer.stage_result_sequence(++last_result_seq)
          outbox_writer.append(last_result_seq, QueryResult)
          live_results_writer.apply_diffs(diffs)
Lib:    session_control.commit() [Atomic DB Commit]

Lib:    { query_output.write() → update QueryOutputState }
Lib:    dispatch_query_result(...) // into live channel
```
If the event fails, `session_control.rollback()` drops all CF writes, leaving memory and DB state intact.

## 5. Dual Bootstrap APIs

`Query` trait extended to replace `as_any()` reflection hacks:

```rust
pub trait Query: Send + Sync {
    // ... existing ...
    async fn fetch_snapshot(&self) -> SnapshotResponse;
    async fn fetch_outbox(&self, after_sequence: Option<u64>) -> Vec<QueryResult>;
}
```

- **`fetch_snapshot`**: Returns `state.results.values()` and `state.as_of_sequence`.
- **`fetch_outbox`**: Returns `state.outbox` entries logically filtered by `> after_sequence`.

## 6. API & Structural Changes

**`ResultDiff`** gains `row_signature: u64` derived from core evaluation bindings context.
**`QueryResult`** gains `sequence: u64` natively.

**Writer Traits** implemented by index plugins:
```rust
pub trait OutboxWriter: Send + Sync {
    fn append(&self, sequence: u64, result: &QueryResult) -> Result<(), IndexError>;
    fn read_all(&self) -> Result<Vec<QueryResult>, IndexError>; // Startup only
    fn clear(&self) -> Result<(), IndexError>;
}
pub trait LiveResultsWriter: Send + Sync {
    fn apply_diffs(&self, diffs: &[ResultDiff]) -> Result<(), IndexError>;
    fn read_snapshot(&self) -> Result<HashMap<u64, Value>, IndexError>; // Startup only
    fn clear(&self) -> Result<(), IndexError>;
}
```

## 7. Persistent vs Volatile Mechanics

Evaluated once via `index_factory.is_volatile(&storage_backend)`:
- **`persistence_enabled = true`**: Storage namespaces written inside transaction. `QueryOutputState` restored from disk.
- **`persistence_enabled = false`**: Core internal transactions only. Zero disk I/O. `QueryOutputState` starts empty and resets to 0 on restart.
